# Правила ведення репозиторію

Цей документ визначає основні правила та стандарти, яких необхідно дотримуватися під час роботи з цим репозиторієм. Дотримання цих правил забезпечує консистентність, читабельність та високу якість кодової бази та документації.

## 1. Ведення журналу змін (Changelog)

- **Обов'язковість:** Усі зміни, що впливають на функціональність, виправляють помилки або додають нові можливості, мають бути відображені у файлі `CHANGELOG.md`.
- **Формат:** Журнал змін ведеться відповідно до стандарту [Keep a Changelog](https://keepachangelog.com/uk/1.0.0/).
- **Структура:** Кожен запис має бути доданий до секції `[Не випущено]` та містити одну з наступних категорій: `Додано`, `Змінено`, `Виправлено`, `Видалено`.
- **Ясність:** Опис змін має бути коротким, але інформативним, зрозумілим як для розробників, так і для користувачів.

## 2. Оновлення та актуалізація документації

- **Актуальність:** Документація повинна завжди відображати поточний стан проекту. Будь-які зміни в коді, що стосуються архітектури, функціональності або процесу збирання, повинні супроводжуватися відповідними оновленнями в документації.
- **Відповідальність:** Розробник, який вносить зміни, несе відповідальність за оновлення всіх пов'язаних документів, включаючи:
  - `STRUCTURE.md`: при зміні структури проекту.
  - `README.md`: при додаванні/зміні ключових функцій.
  - `Docs/`: під час проведення досліджень або додавання технічних оглядів.
- **Повнота:** Документація має бути достатньо повною, щоб новий учасник проекту міг швидко увійти в курс справи.

## 3. Правила спільної роботи та управління гілками

Для забезпечення стабільності основної гілки та організації паралельної роботи використовується наступна модель розгалуження:

- **Основна гілка (`main`):** Гілка `main` містить тільки стабільний, протестований та готовий до використання код. Прямі коміти в `main` заборонені.

- **Функціональні гілки (`feature/`):**
  - Для кожного нового функціоналу, виправлення помилки або великої зміни створюється окрема гілка.
  - Ім'я гілки має відображати її призначення, наприклад: `feature/new-plugin-name`, `fix/login-bug`.
  - Гілка створюється від актуальної версії `main`.

- **Pull Request (PR):**
  - Після завершення роботи у функціональній гілці створюється Pull Request до гілки `main`.
  - В описі PR необхідно чітко викласти суть змін, їх мотивацію та, за необхідності, додати скріншоти або логи.
  - Кожен PR має пройти Code Review від щонайменше одного іншого учасника проекту.

- **Code Review:**
  - Мета Code Review — покращення якості коду, пошук потенційних помилок та забезпечення відповідності стандартам проекту.
  - Рев'юер повинен залишати конструктивні коментарі та пропозиції.
  - Автор PR повинен внести всі необхідні виправлення перед злиттям.

- **Злиття (Merge):**
  - Злиття PR в `main` виконується тільки після успішного проходження Code Review та всіх автоматичних перевірок (якщо вони налаштовані).
  - Переважний метод злиття — `Squash and Merge`, щоб зберегти історію комітів у `main` чистою та лаконічною.
